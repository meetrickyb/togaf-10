    Phase H: Architecture Change Management

    Phase H ensures the architecture remains relevant by managing changes, monitoring implementation, and continuously improving the architecture practice.

    Purpose

    Phase H provides ongoing governance of the architecture after implementation: - Monitor changes in the enterprise and technology landscape - Assess impact of changes on architecture - Manage architecture change process - Maintain architecture lifecycle - Ensure architecture stays aligned with business

    This phase ensures the architecture is a living asset, not a static document.

    Objectives

    Objective Description
    Monitor external changes Track technology, business, and regulatory changes
    Monitor implementation Ensure ongoing compliance with architecture
    Assess change impact Evaluate how changes affect architecture
    Manage change requests Process and prioritize architecture changes
    Update architecture Make necessary revisions to architecture
    Communicate changes Inform stakeholders of architecture updates
    Maintain architecture capability Continuously improve EA practice

    Inputs

    Input Source
    Architecture Contract (signed) Phase G
    Compliance Assessments Phase G
    Implementation and Migration Plan Phase F
    Architecture Definition Document All architecture phases
    Architecture Requirements Specification All architecture phases
    Change Requests Phase G, stakeholders, monitoring
    Organizational Model for EA Preliminary Phase
    Architecture Repository Ongoing updates

    Steps

    1. Establish Value Realization Process

    • Define how benefits will be measured
    • Set up monitoring and reporting mechanisms
    • Establish baseline metrics
    • Create value realization dashboard

    2. Deploy Monitoring Tools

    • Implement architecture monitoring capabilities
    • Set up compliance tracking systems
    • Enable change detection mechanisms
    • Create alerting for significant deviations

    3. Manage Risks

    • Monitor identified implementation risks
    • Identify new risks from changes
    • Update risk register
    • Execute mitigation strategies

    4. Provide Analysis for Architecture Change Management

    • Assess impact of proposed changes
    • Evaluate alignment with business strategy
    • Consider technology trends
    • Analyze cost and benefit of changes

    5. Develop Change Requirements

    • Capture change requests from stakeholders
    • Document business justification
    • Define scope of changes
    • Prioritize based on value and impact

    6. Manage Governance Process

    • Convene Architecture Board regularly
    • Review and approve changes
    • Monitor compliance
    • Make governance decisions

    7. Activate the Process to Implement Change

    • Determine which ADM phase(s) to invoke
    • Initiate new architecture cycle if needed
    • Update architecture incrementally if appropriate
    • Manage architecture version control

    Outputs

    Output Description
    Architecture Updates Revised architecture documents and artifacts
    Change Requests Approved changes triggering new ADM cycles
    Statement of Architecture Work (updated) For new architecture initiatives
    Architecture Contract (new) For new implementation projects
    Compliance Assessments Ongoing compliance monitoring results

    Change Drivers

    Phase H responds to three types of changes:

    1. Business Changes

    Change Type Example Impact
    Strategy shift New market entry May require new capabilities
    Organizational change Merger or acquisition Integration architecture needed
    Process change New business model Business architecture revision
    Regulatory change New compliance requirement Data and security updates

    2. Technology Changes

    Change Type Example Impact
    New technologies Cloud-native platforms Technology architecture opportunities
    End of life Vendor discontinuing product Technology replacement needed
    Security threats New vulnerabilities Security architecture updates
    Performance issues Scalability problems Infrastructure changes

    3. Architecture Improvements

    Change Type Example Impact
    Lessons learned Better patterns discovered Standards and reference models update
    Technical debt Accumulated shortcuts Remediation projects
    Optimization Cost reduction opportunities Rationalization initiatives
    Innovation New architecture approaches Capability enhancement

    Change Request Management

    Change Request Process

    1. Request Submitted
    2. Stakeholder identifies need for change
    3. Change request form completed

    4. Business justification provided

    5. Initial Assessment

    6. Architecture team reviews request
    7. Quick feasibility check

    8. Categorize by type and urgency

    9. Impact Analysis

    10. Assess effect on current architecture
    11. Identify affected domains and stakeholders
    12. Estimate effort and cost

    13. Evaluate benefits and risks

    14. Prioritization

    15. Compare with other change requests
    16. Consider business value and urgency
    17. Assess resource availability

    18. Place in change backlog

    19. Architecture Board Review

    20. Present high-priority changes
    21. Discuss implications and alternatives
    22. Make go/no-go decision

    23. Allocate resources if approved

    24. Implementation

    25. Determine approach (new ADM cycle vs. incremental update)
    26. Execute change
    27. Update architecture repository
    28. Communicate to stakeholders

    Change Request Categories

    Category Handling
    Simplification Architecture improvement opportunity
    Incremental Small refinement
    Re-architecting Significant change
    Emergency Critical urgent fix

    Architecture Monitoring

    What to Monitor

    Area Monitoring Focus Frequency
    Implementation progress Project delivery vs. roadmap Monthly
    Benefits realization Actual vs. expected value Quarterly
    Technology landscape Emerging technologies, vendor changes Ongoing
    Business environment Strategy, organization, market changes Quarterly
    Architecture compliance Conformance to standards Per project
    Technical debt Accumulation of shortcuts and compromises Quarterly
    Risks and issues Threats to architecture objectives Monthly

    Monitoring Mechanisms

    Mechanism Purpose
    Dashboards Real-time view of architecture health
    Compliance audits Periodic assessment of adherence
    Stakeholder surveys Satisfaction and concerns
    Technology watch Horizon scanning for trends
    Metrics and KPIs Quantitative performance measures
    Post-implementation reviews Lessons from completed projects

    Architecture Lifecycle Management

    Architecture Versioning

    Manage architecture as a versioned asset:

    Version Type When to Use Example
    Major Significant re-architecting 1.0 → 2.0 (cloud migration)
    Minor Incremental enhancements 1.1 → 1.2 (new integration pattern)
    Patch Small corrections 1.1.0 → 1.1.1 (standard clarification)

    Architecture States

    State Description Duration
    Current Actively implemented and maintained Until superseded
    Transition Being phased in or out Migration period
    Baseline Reference point for changes Historical record
    Target Planned future state Until implemented
    Sunset Being decommissioned Retirement period
    Retired No longer in use Archived

    Triggering New ADM Cycles

    Phase H determines when to launch a new ADM cycle:

    Triggers for New Cycle

    Trigger Description Example
    Scheduled refresh Regular architecture review Annual or bi-annual cycle
    Significant change Major business or technology shift Digital transformation, merger
    Accumulated changes Many small changes need reconciliation Quarterly incremental updates
    New initiative Major project or program New product line, market entry
    Architecture maturity Capability advancement Move from segment to enterprise architecture

    Cycle Scope Options

    Option Description When to Use
    Full ADM cycle All phases from Preliminary through H Major transformation, new architecture
    Partial cycle Selected phases (e.g., B, C, D, E, F) Update specific domains
    Phase iteration Repeat single phase Refine one area (e.g., re-do Phase B)
    Incremental update Direct architecture edits Minor corrections, small enhancements

    Architecture Governance Board

    The Architecture Board operates continuously in Phase H:

    Board Responsibilities

    Responsibility Description
    Review change requests Assess and approve architecture changes
    Monitor compliance Oversee adherence to architecture
    Resolve exceptions Decide on dispensation requests
    Set priorities Determine architecture work priorities
    Allocate resources Assign architecture capacity
    Escalate issues Raise unresolved matters to executive level

    Board Meeting Cadence

    Meeting Type Frequency Purpose
    Regular meetings Monthly or bi-monthly Routine governance decisions
    Project reviews Per governance checkpoint Compliance assessments
    Strategic sessions Quarterly Architecture direction and planning
    Emergency meetings As needed Critical issues requiring immediate decision

    Continuous Improvement

    Phase H drives architecture practice maturity:

    Improvement Areas

    Area Activities
    Standards and patterns Update based on lessons learned
    Processes Streamline ADM application
    Tools Enhance architecture tools and repository
    Skills Train architects, develop competencies
    Stakeholder engagement Improve communication and collaboration
    Metrics Refine measurement of architecture value

    Architecture Maturity Assessment

    Periodically assess EA capability maturity:

    Maturity Level Characteristics
    Level 0: None No formal EA practice
    Level 1: Initial Ad-hoc architecture, project-based
    Level 2: Developing Some standards, emerging governance
    Level 3: Defined Documented processes, consistent application
    Level 4: Managed Measured performance, proactive governance
    Level 5: Optimizing Continuous improvement, strategic asset

    Relationship to Other Phases

    Phase Relationship
    Phase G Provides lessons learned and change requests to Phase H
    Preliminary Phase H may trigger update to architecture framework
    Phase A Phase H launches new architecture cycles starting at Phase A
    All phases Phase H determines which phases to invoke for changes
    Requirements Mgmt Continuous requirement management operates alongside Phase H

    Phase H is continuous — it never really ends, transitioning to new ADM cycles as needed.

    Exam Tips

    • Phase H provides ongoing governance and change management for architecture
    • Monitors three types of changes: Business, Technology, Architecture improvements
    • Change requests processed, prioritized, and routed appropriately
    • Phase H determines when to launch new ADM cycles
    • Architecture Board operates continuously to govern architecture
    • Architecture managed as versioned, living asset
    • Monitoring includes: implementation progress, benefits realization, technology trends
    • Change categories: Simplification, Incremental, Re-architecting, Emergency
    • Phase H feeds back to Preliminary (framework updates) or Phase A (new cycles)
    • Continuous improvement of architecture capability and maturity
    • Architecture states: Current, Transition, Baseline, Target, Sunset, Retired
    • Phase H ensures architecture stays aligned with business over time
